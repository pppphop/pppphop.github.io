BUAA OO U3总结博客

一、对 JML 与规格驱动开发的理解

Unit3 的核心变化，是开发方式从“看自然语言描述写程序”转向“阅读接口规格并实现契约”。JML 给我的最大感受是：它把方法的可见行为拆成了几个相对清晰的部分，包括前置条件 requires、后置条件 ensures、副作用范围 assignable、正常行为 normal_behavior、异常行为 exceptional_behavior 以及用于表达旧状态的 \old。这些内容本质上是在回答三个问题：调用者什么时候可以调用？实现者调用后必须保证什么？方法执行过程中允许改动哪些状态？

相比自然语言，JML 的优势在于边界更明确。例如一个查询方法如果标注为 pure 或 assignable \nothing，就不能为了实现方便去偷偷修改可观察状态；一个删除方法如果在 ensures 里说明了删除对象、保留对象和返回值的关系，就不能只凭“看起来删掉了”来判断正确。在本单元中，我逐渐形成的做法是：先把规格翻译成状态变化表，再决定数据结构，而不是先写容器再让规格去适配实现。

我也认识到，规格中的 model 字段不等同于必须照抄的实现字段。接口中可能用数组、集合或者序列来描述外部可见状态，但实现时可以使用 HashMap、HashSet、LinkedList、缓存变量等结构，只要最终满足规格所描述的可观察行为即可。因此，规格驱动开发并不是“机械翻译 JML”，而是“以 JML 为准绳选择实现策略”。

另外，JML 中的异常规格也非常重要。三次作业中，很多异常类本身没有 JML，需要结合指导书和官方包文档理解。我的体会是：异常检查顺序、异常参数、异常前状态保持，都是规格的一部分。对于带有事务性质的方法，例如投币、购买勋章、评论清理，必须先完成所有合法性检查，再修改多个对象的状态，否则很容易出现“前半部分已经改了，后半部分抛异常”的不一致问题。

二、JUnit 测试经验总结

本单元每次都要求针对一个指定方法写 JUnit 测试。我最大的收获是，单元测试不能只测返回值，还要测完整契约。

第一次作业测试 queryMutualFollowingSum 时，我没有直接相信被测方法本身，而是在测试中用双重循环重新计算互相关注对数，作为独立 oracle。为了覆盖常见错误，我构造了空图、单用户、单向关注、单个互关、非对称发散图、单向环、复杂互关图和取消关注后恢复等场景。这样可以发现把“总边数/2”误认为互关数、只统计入度/出度、取消关注后缓存不更新等错误。

第二次作业测试 clean_spam_comments 时，我更加注意 assignable 和序列保持。我在调用前保存评论 ID、评论内容、播放数、点赞数、转发数、投币数、视频类型、上传者 ID，以及相邻视频的评论区快照。调用后不仅检查返回的删除数量和最大关键字出现次数，还检查未删除评论的顺序是否保持、其他字段是否被错误修改、其他视频是否受到影响。针对字符串匹配，我专门设计了 a.b、a*b 这类容易被正则误伤的用例，以及 ababa 中关键字 aba 的重叠出现，用来避免把 JML 的字面 substring 匹配误写成正则匹配或非重叠匹配。

第三次作业测试 recommend_Nth_up 时，我的思路是把排序规则独立出来。测试中需要覆盖非法 rank、无视频、候选 UP 主不足、分数相同按 ID 取小、排除自己和已经关注的人、连续调用不改变用户状态等场景。推荐类方法看上去只是“返回一个 ID”，但它隐含了候选集合筛选、打分、排序、异常和纯净性多个方面，任何一个环节漏测都可能放过错误实现。

总结下来，JUnit 测试最有效的模式有三点：第一，测试 oracle 要尽量独立于正式实现，避免复制同一个 bug；第二，要对调用前后的状态拍快照，尤其是纯查询方法和只允许局部修改的方法；第三，边界数据比大规模随机数据更容易发现规格理解错误，例如空集合、重复 ID、相同分数、重复转发、重叠字符串、并列最大值和 tie-break。

三、三次作业的迭代过程

1. 第一次作业：社交网络与基础视频推送

第一次作业的主要任务是实现 User、Network、Video 三个类，维护用户、视频、关注关系、粉丝关系、接收视频列表，以及互相关注数、最短路径等查询。

我的实现中，Network 维护了 users 和 videos 两个 HashMap，并额外维护 outEdges 和 inEdges 两张邻接表，分别表示“某人关注了哪些人”和“某人被哪些人关注”。这样做的原因是，关注图既要支持关注/取消关注，也要支持最短路查询和互相关注统计。如果完全依赖 User 内部容器反复遍历，会在高频查询时产生不必要的复杂度。

对于 queryMutualFollowingSum，我没有在每次查询时重新双重循环统计，而是在 followUser 和 unfollowUser 中动态维护 mutualFollowingSum。当新建边 id1 -> id2 时，如果反向边 id2 -> id1 已经存在，则互关数加一；删除边时同理减一。因此查询可以做到 O(1)。对于 queryShortestPath，我使用 BFS，在有向关注图上查找最短路径。

User 中使用 HashMap 管理 following 和 followers，用 LinkedList 管理收到的视频列表，便于新上传视频头插；同时维护年龄段计数数组 ageSegments，使粉丝年龄比例查询不必每次遍历粉丝集合。

2. 第二次作业：互动系统、硬币体系与评论清理

第二次作业在第一次基础上加入了硬币、观看历史、点赞记录、投币、转发、评论、粉丝勋章、视频热度和最长年龄下降序列等功能。相比第一次，第二次最大的挑战是多对象状态需要同时更新。例如投币成功时，投币用户的硬币数减少，视频的投币数增加，UP 主的硬币数增加，UP 主的贡献者记录也要更新；这些更新必须在所有异常检查之后统一发生。

我的 User 新增了 watchedVideoIds、likedVideoIds、medalUpIds 和 contributorMap。对于最佳贡献者，我没有每次查询都遍历 contributorMap，而是在每次收到投币时维护 bestContributorId 和 maxContribution，并在贡献值相同的时候按较小 ID 更新。

Video 新增了分区类型、播放量、点赞数、转发数、投币数，以及评论 ID 和评论内容两个平行 ArrayList。为了快速判断评论 ID 是否重复，我额外维护 commentIdSet。评论清理使用迭代器同步删除 ID 和内容，并返回删除数量和被删评论中关键字最大出现次数。

此外，第二次开始我明显感受到“规格正确”和“复杂度可接受”是两件事。比如视频分区查询如果每次遍历所有视频也能写对，但在强测中可能超时，所以我维护了 typeVideos。最短路查询也增加了缓存，但所有会改变关注图的操作都会清空缓存，避免读到过期结果。最长年龄下降序列则使用 DFS + DP 缓存，并在可能改变结果的图操作后失效。

3. 第三次作业：智能推荐与结构拆分

第三次作业继续扩展系统，加入推荐视频、推荐第 N 个 UP 主、查询最有影响力 UP 主、查询用户画像、查询全局最佳贡献者等功能。此时如果继续把所有逻辑塞进 Network，类会变得过长且难以定位问题。因此我把代码拆分为三个主要部分：Network 作为接口门面，负责异常检查和对外方法；SocialGraph 负责关注图、互关数、最短路和最长下降序列；RecommendationEngine 负责推荐和贡献者统计。

在 User 中，我新增了 typeCounts 来记录用户在各分区观看的视频数量，并维护用户上传的视频和分区影响力。观看视频时，不仅要从 receivedVideos 中删除该视频，还要加入观看历史，并增加对应分区的观看计数。推荐视频时，视频得分可以理解为视频热度与用户对该分区兴趣的乘积；推荐 UP 主时，则根据用户兴趣和 UP 主分区影响力进行加权求和。

性能方面，第三次我做了两点优化。第一，UP 主影响力不再每次通过遍历其所有上传视频求和，而是在观看、点赞、取消点赞、投币、转发等热度变化点增量维护。第二，recommendNthUp 不再对所有候选人完整排序，而是用大小为 rank 的优先队列保留当前前 rank 名，把复杂度从全排序降到和 rank 相关的堆维护。这也让我认识到，JML 中的 \sum、\max、排序式描述往往表达的是数学结果，而不是要求实现必须用同样朴素的枚举方式。

四、如何发现迭代中的方法和容器变化

我主要使用三步法发现变化。

第一步是看指导书和接口中的 Modify、新增方法和构造方法变化。例如第二次 Video 构造方法新增 type 参数，User 新增 coins、watchedVideos、likedVideos、medals、contributors、contributions 等状态；第三次 User 又新增 typeCounts 和 videos，Network 新增智能推荐相关方法。

第二步是看 JML 中新增的 ensures、assignable 和异常规格。如果某个后置条件开始引用一个新的状态量，就说明实现中需要新增容器或缓存。例如 hasWatchedVideo、hasLikedVideo、containsComment、getInterest、getInfluence 等方法都对应着新的可观察状态。

第三步是用编译错误和回归测试辅助确认。方法签名变化会直接导致编译失败；行为变化则要靠旧测试回归和新增测试发现。例如第二次保留第一次关注图功能，第三次保留前两次所有指令，这要求新增功能不能破坏原有容器不变式。

五、如何发现性能瓶颈

说实话，其实写到那个for自己就能意识到这里性能可能有点问题，但是严谨来说，我的性能瓶颈排查主要来自两个方面：规格复杂度和调用频率。

从规格复杂度看，只要 JML 中出现嵌套量词、全局 \sum、最大值、排序、路径存在性，就要警惕直接翻译造成高复杂度。例如互相关注数如果每次查询都双重遍历用户是 O(n²)，最短路径如果每次都 BFS 在大量重复查询下可能很慢，影响力如果每次都遍历所有视频也会在推荐场景中被放大。

从调用频率看，强测通常会把查询类指令大量重复调用，所以适合用动态维护或缓存。我的处理方式是：互关数、年龄段人数、最佳贡献者、UP 主影响力采用增量维护；最短路径和最长下降序列采用缓存，但在图变化时失效；分区视频集合单独维护，避免每次查询遍历所有视频；推荐第 N 个 UP 主使用优先队列，避免全量排序。

这也带来一个原则：缓存不能破坏规格。对于纯查询或只允许修改特定对象的方法，即使内部用了缓存，也必须确保不会改变任何外部可观察状态，并且在相关状态变化时及时失效。

六、Bug 与原因分析

本单元的 Bug 分析可以分为三类：提交前自测发现的问题、互测中发现的典型实现问题，以及性能风险与复杂度问题。

6.1 提交前自测发现的问题

  1. 对象相等语义理解错误
    • 第一次作业早期版本中，Video.equals 同时比较 id 和 uploaderId。
    • 修正思路：视频在网络中的唯一身份由 videoId 决定，规格容器的不变式和方法如 containsVideo 都围绕 ID 判断。最终 Video.equals 只比较 getId()。
  2. 异常参数混淆
    • 第二次作业中，购买勋章时如果用户已经拥有该 UP 主勋章，曾错误抛出 DuplicateMedalException(userId, videoId)。
    • 修正思路：勋章绑定 UP 主而非视频，正确参数应为 DuplicateMedalException(userId, upId)。
  3. 评论清理边界匹配错误
    • 在 clean_spam_comments 测试中，字面关键字匹配和非重叠匹配理解不当，导致 a.b、a*b、ababa 等边界用例出错。
    • 修正思路：按规格要求的字面匹配实现 oracle，确保重叠出现的关键字被正确统计和删除。
  4. 异常顺序与事务一致性
    • 投币、购买勋章等方法中，若在检查异常前已修改部分对象状态，可能导致状态不一致。
    • 修正思路：先完成所有异常检查，再统一更新多个对象状态，确保事务一致性。

6.2 互测中发现的典型问题

  1. hw9 双向 BFS 写错
    • 部分实现没有正确维护双向搜索的前后关系，导致最短路径查询返回错误或陷入死循环。
  2. hw11 UpRecommender 排序溢出
    • 有实现使用 return user1.getId() - user2.getId(); 比较用户 ID，由于 int 溢出，排序可能出错。
  3. 特殊标识冲突
    • 有实现用 -1 作为标识值，但 -1 本身也是合法用户 ID，导致逻辑异常。
  4. 异常输出顺序错误
    • 部分实现能抛出正确异常，但输出顺序不符合要求，影响测试判定。

6.3 性能问题与复杂度风险

  1. 重复遍历容器
    • 某些实现对视频列表、关注图、贡献者映射每次查询都全量遍历，容易在强测中超时。
  2. 缓存与增量更新不当
    • UP 主影响力、互关数、年龄段统计等若不增量维护，频繁查询时复杂度被放大。
  3. 排序与全量扫描
    • 推荐第 N 个 UP 主或推荐视频，如果对所有候选完整排序而不是使用优先队列或部分排序，复杂度随用户数量急剧增加。
  4. 最长下降序列和最短路径缓存
    • 缓存若未在关注图变化时失效，可能返回过期结果，违反纯查询或副作用约束。

七、大模型使用体会

本单元我更适合把大模型当作“辅助阅读规格和生成测试思路”的工具，而不是直接生成最终代码的工具。它的优势在于可以快速把长 JML 翻译成自然语言、列出可能的边界条件、提醒某个方法需要检查哪些副作用，也可以帮助生成 JUnit 用例框架。

但大模型的风险也很明显。第一，它容易给出“看起来正确”的朴素实现，比如按 JML 的数学表达直接遍历所有对象，却忽略强测中的性能压力。第二，它不一定知道课程官方包中异常参数、输出格式和扩展语义的细节。第三，它可能忽视容器架构，导致新增功能时不断在原类中堆字段和方法，而不是抽象出 SocialGraph、RecommendationEngine 这样的组件。

因此我认为比较安全的使用方式是：让大模型帮助列测试清单、解释某段 JML、对照 requires/ensures/assignable 检查遗漏；最终实现仍然要自己结合官方接口、复杂度和评测限制判断。对于单元测试，可以让大模型先列边界场景，再由自己写独立 oracle 和状态快照检查。

另外，大模型对于时间复杂度的优化很有用，可以严防TLE。

八、JML“击鼓传花”研讨课感悟

本次研讨课的游戏名为“规格传声筒”，规则类似“击鼓传花”或“传话游戏”。每组 6 到 7 人，每个人先写一个带有边界条件的初始方法签名与业务场景，然后依次经历 自然语言规格 → JML 规格 → 自然语言规格 → JML 规格 → 自然语言规格 → 最终 JML 规格 的多轮环状传递。传递过程中禁止使用 AI，所有回合结束后，再对比第一轮的初始自然语言规格和最后一轮的 JML 规格是否逻辑等价。评分重点包括逻辑同构、边界遗失、过度设计和业务畸变等。规格传声筒

这次活动让我非常直观地感受到，规格在传递过程中很容易发生“语义漂移”。最开始的自然语言需求可能只是一个很小的方法，例如查询集合、更新状态、处理图关系或统计某类对象，但经过几轮 NL 和 JML 的相互翻译后，原来的边界条件、异常行为、修改范围甚至业务目标都有可能被改变。比如原本要求处理 null、空集合或重复元素，传到后面可能这些边界就消失了；原本只是查询方法，后面可能漏写 assignable \nothing；原本有异常行为，后面可能只剩下正常行为；原本的 tie-break 规则，也可能在传递中被忽略。

我认为这个活动最有意义的地方在于，它暴露了自然语言和形式化规格之间的差距。自然语言的优点是易读，但很容易产生歧义；JML 的优点是精确，但写规格的人如果没有完全理解业务含义，也会把错误理解“形式化”下来。一旦错误被写进 JML，后续实现者往往会认为它就是正确契约，从而导致实现偏离最初需求。因此，形式化并不等于天然正确，形式化之前必须先保证需求理解正确。

游戏中的几类扣分点也很贴近真实开发。所谓“边界遗失”，对应实际项目中常见的空容器、非法输入、重复对象、不可达路径等问题；“过度设计”则对应实现者或规格编写者加入了原需求没有要求的限制，例如额外限制数组长度、ID 范围或调用顺序；“业务畸变”更严重，意味着方法的核心功能在多次转述后已经变成了另一个问题。这些问题在本单元三次作业中也经常出现，例如异常顺序写错、用非法标识值代替合法 ID、把推荐排序规则理解错，或者忘记某个方法是纯查询。

这次研讨课也让我重新理解了 JML 在团队协作中的价值。JML 不只是为了通过评测，也不是为了把代码写得更“形式化”，它真正的作用是帮助团队成员共享同一个契约。一个方法到底允许修改哪些对象、什么时候抛异常、返回值和旧状态有什么关系、并列情况下如何排序，这些内容如果只靠口头说明，很容易在多人协作中被误解；如果用 JML 明确下来，后续写代码、写测试和做评审时就都有了统一依据。

同时，我也认识到，团队合作中不能只把 JML 当作最终结果，而应该把规格编写过程本身纳入评审。比较合理的流程应该是：先由需求提出者写出自然语言说明，明确正常行为、异常行为、边界条件和副作用；再由规格编写者写出 JML；然后由其他成员对照自然语言和 JML 检查是否等价。尤其要重点检查 requires、ensures、assignable、signals、\old 和量词表达式是否遗漏或过度约束。只有经过这种双向检查，JML 才能真正减少沟通成本，而不是制造新的误解。

从本单元作业的角度看，“规格传声筒”也提醒我，写测试时不能只测试自己脑中的需求，而要测试规格中真正写出来的契约。如果规格中要求纯方法，就要检查调用前后状态不变；如果规格中要求异常时 assignable \nothing，就要检查抛异常后对象没有被部分修改；如果规格中有排序规则，就要专门构造并列情况；如果规格中有边界条件，就要测试空集合、重复元素、不可达路径、非法参数等情况。也就是说，测试本身也是对规格理解是否准确的一次验证。

总的来说，这次研讨课让我意识到，软件开发中的很多 bug 并不是因为不会写代码，而是因为需求在传递、翻译和实现过程中被悄悄改变了。JML 的价值就在于尽量压缩这种“我以为”的空间，让需求、规格、实现和测试围绕同一个契约展开。以后在多人组队编程中，我会更加重视规格文档的唯一性、边界条件的显式记录、异常行为的完整描述，以及每次需求修改后的同步评审。

九、总结

Unit3 让我把面向对象设计从“类和方法的组织”推进到“契约、状态和复杂度的组织”。第一次作业让我熟悉了用规格描述社交图；第二次作业让我意识到事务性操作和副作用控制的重要性；第三次作业则让我开始主动拆分模块、维护缓存并从复杂度角度理解 JML。

最终我对规格驱动开发的理解是：JML 给出的是程序外部可观察行为的数学契约，真正的实现既要满足这个契约，也要在数据结构和复杂度上适应评测规模。好的实现不是简单把 JML 翻译成循环，而是在不改变规格语义的前提下，选择合适的容器、缓存和模块划分。好的测试也不是只看输出，而是把前置条件、后置条件、副作用和异常路径都纳入验证。这个单元之后，我会更加重视在编码前先读规格、在迭代中维护变化表、在提交前用测试检查契约完整性。
