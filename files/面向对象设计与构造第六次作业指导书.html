面向对象设计与构造第六次作业指导书

提示：

  • 本次作业官方包已经放在 http://gitlab.oo.buaa.edu.cn/2026_oo_public/officialpackage/u2_hw6
  • 本次作业的新增内容有：
    • 取消“乘客请求指定电梯”约束，需要自行设计电梯分配策略。
    • 新增“临时检修”请求。
    • 修改RECEIVE约束部分。
    • 修改乘客请求格式；修改测试数据约束。
  • 具体内容见指导书对应部分。

第一部分：训练目标

  • 掌握多线程的基本知识，如线程的创建、线程的互斥与协作、wait/notify机制等。
  • 掌握生产者-消费者模式，能设计简单的多线程程序。

第二部分：题目概况

本单元以某公司大楼的目的选层电梯系统为背景。

目的选层电梯系统的运行流程：

  1. 进入电梯前，乘客输入目标楼层；
  2. 系统根据设定的策略指定最合适的电梯来服务，将结果告知乘客；
  3. 乘客进入指定电梯后，无需再次按楼层按钮，电梯会自动将其送至目标楼层。

在三次迭代作业中，你需要根据需求与正确性约束，设计多线程程序来模拟该电梯系统的运行。

系统从标准输入中读入乘客请求信息（起点层、终点楼层），请求调度器会根据此时电梯运行状态将乘客请求合理分配给某部电梯。被分配请求的电梯会经过上下行，开关门，乘客进入/离开电梯等动作将乘客从起点层运送到终点层。

请求的输入通过课程组提供的输入接口来定时投放请求，你需要调用课程组提供的接口进行输入输出。接口的具体细节见第四部分-输入/输出接口和相关文档。

单部电梯的运行策略与电梯系统的调度策略均可任意指定。也就是说，在任意时刻，电梯的上下行动、开关门操作都可以自定义，只要保证在电梯系统运行时间不超过题目要求时间上限的前提下将所有的乘客送至目的地即可。电梯运行的具体介绍见第三部分。

第三部分：电梯运行（改动）

功能概述（改动）

对于每个乘客请求，需要调度电梯将其完成，并将必要的运行信息通过输出接口进行输出。

具体而言，你需要控制电梯上下行，开关门的动作以及控制乘客进出电梯来将乘客从起点层运送到终点层。在控制电梯运行的过程中需要满足所有系统运行约束，如运行速度、电梯载重等，运行约束的具体细节见本部分的系统运行约束一节。

在本次作业中：

  • 我们不再为每个乘客请求指定电梯。你需要先通过输出 RECEIVE 信息表示乘客被分派给哪个电梯，然后调度指定的电梯来接送对应乘客。（改动）
  • （新增） 我们新增了临时检修请求。电梯会在正常运行时接收到临时检修请求，你需要合理安排当前未完成的乘客请求，并在限定时间内按规定流程调度电梯完成检修，在完成检修后电梯恢复正常运行（具体见本部分-临时检修一节）。

电梯系统默认参数

  • 可到达楼层：地下B4-B1层，地上F1-F7层，共11层
  • 初始位置：F1层
  • 数量：6部
  • 编号：6部电梯的id依次为1-6
  • 移动速度：0.4s/层
  • 额定载重量：400kg
  • 开关门：开门到关门之间的间隔不小于0.4s

电梯载重说明

在现实情境中，电梯开门时允许轿厢内乘客暂时超过载重限制。当轿厢内乘客总体重超过载重时无法关门，此时会有乘客出电梯，等到符合载重限制才能出发。我们的作业要求与此相同，允许开门时暂时超过载重限制，只需保证关门时轿厢符合载重限制即可。电梯系统的全部6部电梯载重均为400kg。

RECEIVE约束说明（改动）

为避免出现多部电梯接送同一乘客造成资源浪费的情况，我们引入 RECEIVE 约束，要求在乘客被分配给某一电梯后输出相应 RECEIVE 信息。

本次作业不再为乘客请求指定电梯，需要自行设计电梯分配策略。因此需要在 RECEIVE 信息中输出每个乘客被分配给了哪个电梯运送。电梯运行受到 RECEIVE 的约束，简单来说必须保证电梯在做出任何动作时都必须有未完成的 RECEIVE。具体要求见本部分的系统运行约束一节。

临时检修（新增）

为了按需检修电梯系统，有时会派出检修工人来检修电梯。这一过程称为临时检修，临时检修的核心过程主要分为部件检修和验收测试两部分。

检修流程

为了避免歧义，我们将引入检修后的电梯运行模式建模为状态机，电梯初始的默认状态为NORMAL状态。 每次临时检修时，电梯状态都须按下表顺序进行转移 (NORMAL -> ACCEPT -> REPAIR -> TEST -> NORMAL)。在每个状态中，电梯都依次需要完成“电梯行为流程”的每一项。每个状态行为流程的最后一条都是状态转移标志(用 * 表示)，即电梯在完成这一条行为流程的时刻进入下一状态。

电梯状态 转移到此状态的标志 电梯行为流程 电梯行为额外约束
NORMAL (默认状态) 1. 初始状态 2. 程序输出 MAINT-END (上一次检修结束) 1. 正常运行，直到收到检修指令 *2. 收到检修指令，程序输出 MAINT-ACCEPT (官方包自动输出) 无
REP_ACCEPT (收到检修指令状态) 程序输出 MAINT-ACCEPT (输入检修指令后，官方包自动输出) 1. 电梯运行到F1层，准备接受检修 2. 轿厢内所有普通乘客都下电梯（无普通乘客则省略） *3. 检修工从F1层进入轿厢中，然后电梯关门，随后程序输出部件检修开始信息 MAINT1-BEGIN 1. 在检修工进入电梯时，电梯内必须没有任何普通乘客 2. 从检修工进入电梯到程序输出 MAINT1-BEGIN 期间，电梯不允许移动、开门、进出乘客（只允许关门一次） 3. 输出 MAINT1-BEGIN 时，电梯必须处于REP_ACCEPT状态，必须处于关门状态 4. 见本小节-额外说明-REP_ACCEPT状态特殊约束
REPAIR (部件检修状态) 程序输出 MAINT1-BEGIN (部件检修开始信息) 1. 由工人检修电梯部件，电梯等待至少 $T_{stop} = 1s$ *2. 程序输出验收测试开始信息 MAINT2-BEGIN 1. 在输出 MAINT1-BEGIN 时刻（进入REPAIR状态时刻），电梯此前的所有RECEIVE结束（仅RECEIVE结束，相应乘客请求仍需完成） 2. 该状态下电梯处于静默状态，禁止移动、开关门、进出乘客 3. 该状态下电梯禁止RECEIVE乘客，禁止输出该电梯RECEIVE信息 4. 输出 MAINT2-BEGIN 时，电梯必须处于REPAIR状态
TEST (验收测试状态) 程序输出 MAINT2-BEGIN (验收测试开始信息) 1. 电梯速度调整为0.2s/层 2. 从F1层直接驶向临时检修指令中指定的目标楼层 3. 从目标楼层直接驶回F1层 4. 电梯速度恢复默认值0.4s/层 5. 在F1层，电梯开门并让检修工下电梯，然后电梯关门 *6. 程序输出检修结束信息 MAINT-END 1. 该状态下电梯移动速度为0.2s/层，该状态结束后速度恢复原值 2. 该状态下电梯必须按规定路线移动 3. 该状态下电梯只能在F1层开门，且只能开门关门一次，并必须在这次开关门期间让检修工下电梯 4. 该状态下乘客不能进入电梯 5. 该状态下电梯禁止RECEIVE乘客，禁止输出该电梯RECEIVE信息 6. 输出 MAINT-END 时，电梯必须处于TEST状态

额外说明

REP_ACCEPT状态特殊约束：

在表格中REP_ACCEPT状态的第一项行为是“电梯运行到F1层，准备接受检修”。但我们不强制要求电梯在接收到检修指令后立即前往F1层，也不强制要求电梯在到达F1层后就进入后续行为流程(让所有普通乘客下电梯，让检修工上电梯)。即进入REP_ACCEPT状态时，电梯的运行仍然自由，程序可自行决定电梯何时前往F1接收检修。程序可以先正常运送乘客一段时间，中途可以正常经过F1层，也可以正常在F1层开关门、上下乘客。只要检修工不上电梯，电梯就依然处于REP_ACCEPT阶段。

程序不能无限拖延检修开始时间，必须满足检修时间约束。检修时间约束规定了整个检修流程的最长时间，受此限制电梯必须在一定的时间内完成REP_ACCEPT状态的三项流程，进入下一状态。

检修时间约束：

输出 MAINT-ACCEPT 和 MAINT-END 之间的时间记为检修响应时间 $T{complete}$，$T{complete}$ 不得超过7s。除此之外，我们不对收到 MAINT-ACCEPT 后电梯何时前往F1层接收检修做硬性约束。可以选择在收到 MAINT-ACCEPT 后立即前往F1层，也可以选择先进行一些其他的电梯调度行为再前往F1层。只要满足 $T_{complete} \le 7s$ 的约束即可。

检修工人

  • 检修工人在上下电梯时的输出信息与普通乘客完全一致。并同样遵循相同电梯运行规则（如必须在电梯开门时上电梯、上下电梯时乘客和轿厢需要处于同一层等）。
  • 检修工人的id在检修指令中给出，起点层和终点层均固定为F1，体重不给出（保证每个检修工体重都小于电梯载重限制）。
  • 检修工人上电梯前无需输出 RECEIVE。
  • 检修工人不计入性能分计算。

缩写说明

在本节中输出信息使用了缩写，如 MAINT-ACCEPT，省略了时间戳和电梯ID。完整的输出信息格式见第四部分-输出说明。

RECEIVE说明

在 MAINT-END 时“解除禁止电梯RECEIVE请求的限制”表示从此刻起，电梯可以正常RECEIVE乘客请求。但不会自动恢复在 MAINT1-BEGIN 时强制结束的RECEIVE。也就是说从 MAINT1-BEGIN 时刻起，程序就需要将这些原本RECEIVE给待检修电梯的乘客请求重新进行RECEIVE了。程序既可以让其他正常运行电梯RECEIVE这些乘客，也可以等待当前电梯检修完成后RECEIVE这些乘客。

系统运行约束（改动）

  • 电梯移动约束
    • 电梯在两层楼之间移动应满足移动时间约束。
    • 电梯移动过程中必须处于关门状态。
    • 电梯只能在大楼的楼层范围内移动。
  • 电梯开关门状态约束
    • 初始时所有电梯为关门状态。
    • 程序运行结束时所有电梯必须处于关门状态。
    • 开门状态下才能关门，关门状态下才能开门。
    • 开门到关门之间的间隔不小于0.4s。
  • RECEIVE约束（改动）
    • 在本次作业中，输入请求不再为每个乘客指定电梯，需要电梯系统给乘客分派电梯并通过 RECEIVE 输出。
    • 乘客进入电梯必须在输出该乘客的 RECEIVE 信息之后（注意中途下电梯的乘客也需要重新输出 RECEIVE 信息才可上电梯）。
    • 电梯移动必须在该电梯当前有未结束的 RECEIVE 时才能移动。这里的移动指开始移动，即 $t{arrive时刻} - t{每层移动时间}$ 时刻。
    • 程序不能RECEIVE已经从目标层离开的乘客，不能RECEIVE上一次RECEIVE未结束的乘客。
    • 程序输出某一乘客的 RECEIVE 信息时刻，相应的RECEIVE开始。
    • 程序输出某一乘客下电梯信息时刻，相应的RECEIVE结束（在目标层或中途下电梯均视为RECEIVE结束）。
    • （新增） 电梯部件检修开始时(MAINT1-BEGIN)，该电梯此时所有RECEIVE结束。
    • （新增） 从电梯部件检修开始时刻(MAINT1-BEGIN)到电梯验收测试结束时刻(MAINT-END)，该电梯不允许输出任何RECEIVE信息。
  • 电梯检修约束（新增）
    • 电梯只有在收到检修指令时才能进行检修。
    • 电梯检修过程中必须满足本部分-临时检修中的所有要求。
  • 乘客进出约束
    • 乘客默认不在电梯里，电梯开门状态下才能进出电梯，且只能进入同层的电梯。
    • 乘客只有在电梯内才能出电梯，只有在电梯外才能进电梯。
    • 开门状态下乘客进出电梯不花费时间。
    • 乘客只能进入当前RECEIVE的电梯。
  • 乘客运送约束
    • 系统运行中乘客可以中途下电梯，系统结束时到达目的地即可。
    • 系统结束时必须完成全部乘客请求。
    • 系统结束时不能有乘客困在电梯里。
  • 电梯载重约束
    • 电梯为开门状态时，允许轿厢内乘客体重暂时大于额定载重量。
    • 电梯为关门状态时，须满足轿厢内乘客体重之和小于等于额定载重量。
  • 运行时间约束
    • 程序运行时间不能超过 $T{max}$。公测、强测、互测的 $T{max}$ 规则介绍见第四部分-数据限制。
  • 输出限制
    • 对每个乘客请求，均须输出相应RECEIVE信息。
    • 电梯每次移动后到达新楼层时须输出相应信息。
    • 电梯每次开门、关门时均须输出相应信息。
    • 乘客每次进入、离开电梯时均须输出相应信息。
    • 在临时检修期间，部件检修开始（检修开始）、验收测试开始和验收测试结束（检修结束）时均须输出相应信息。
    • 运行时的任意时刻须满足输出时间戳非减限制。

第四部分：输入/输出说明（改动）

输入/输出接口说明

本单元作业请使用课程组提供的输入输出接口进行输入输出，见本次作业公开仓库。

输入接口提供了若干方法，通过调用方法可以直接获取所需数据对象。输出子程序接受一个字符串，并附上时间戳后再输出。详细的接口定义和使用方法见接口说明文档。

程序的输入输出为实时交互，评测机可以做到在某个时间点投放一定量的输入。不需要考虑由于调用输入输出接口导致的时间上的差异，不需要对输入输出接口进行修改。

输入格式

  • 乘客请求：[时间戳]乘客ID-WEI-体重-FROM-起点层-TO-终点层
  • 检修指令（新增）：[时间戳]MAINT-电梯ID-检修工人ID-验收目标楼层

输出格式

通过调用输出接口的每一次输出将自动附加时间戳于头部。请不要试图伪造时间戳。

  • 电梯接收分配：[时间戳]RECEIVE-乘客ID-电梯ID
  • 电梯到达某一位置：[时间戳]ARRIVE-所在层-电梯ID
  • 电梯开始开门：[时间戳]OPEN-所在层-电梯ID
  • 电梯完成关门：[时间戳]CLOSE-所在层-电梯ID
  • 乘客进入电梯：[时间戳]IN-乘客ID-所在层-电梯ID
  • 乘客离开电梯：
    • 当前楼层是目标楼层：[时间戳]OUT-S-乘客ID-所在层-电梯ID
    • 当前楼层不是目标楼层：[时间戳]OUT-F-乘客ID-所在层-电梯ID
  • 临时检修输出（新增）：
    • 检修指令接受：[时间戳]MAINT-ACCEPT-电梯ID-检修工人ID-验收目标楼层 (由官方包自动输出)
    • 部件检修开始：[时间戳]MAINT1-BEGIN-电梯ID
    • 验收测试开始：[时间戳]MAINT2-BEGIN-电梯ID
    • 验收测试结束：[时间戳]MAINT-END-电梯ID

数据限制（改动）

基本限制（改动）

  • 可输入电梯系统的指令数：1~100条。
  • 每个输入请求具有唯一不重复的id（int范围内正整数）。
  • 乘客体重为 [50, 100] 范围内的正整数。
  • 保证不出现起点层和终点层相同的乘客请求。
  • （新增） 保证检修指令的验收目标楼层 $\in {B2, B1, F2, F3}$。
  • （新增） 保证同一电梯的两条MAINT指令至少间隔8s。

互测数据限制（改动）

第六部分：性能分

在强测时，性能分将由三部分组成：系统运行时间，平均完成时间和系统耗电量。

  • 系统运行时间：$T{run} = \max(T{real}, T_{final})$。
  • 平均完成时间：$T{avg} = \frac{\sum_i (T_i - T{i0})}{N}$
  • 系统耗电量：
    • 每条ARRIVE信息耗电量 $W_{ARRIVE} = 0.4$
    • 每条OPEN/CLOSE信息耗电量 $W{OPEN} = W{CLOSE} = 0.1$
    • 系统的耗电量 $W = W{OPEN}N{OPEN} + W{CLOSE}N{CLOSE} + W{ARRIVE}N{ARRIVE}$
