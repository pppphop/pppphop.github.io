面向对象设计与构造第七次作业指导书

提示：本次作业官方包已经放在 http://gitlab.oo.buaa.edu.cn/2026_oo_public/officialpackage/u2_hw7
本次作业的新增内容有：

  • 新增“双轿厢改造”与“双轿厢回收”
  • 修改 RECEIVE 约束部分
  • 修改测试数据约束；修改“实现提示”部分

具体内容见指导书对应部分。

第一部分：训练目标

  1. 掌握多线程的基本知识，如线程的创建、线程的互斥与协作、wait/notify机制等。
  2. 掌握生产者-消费者模式，能设计简单的多线程程序。

第二部分：题目概况

本单元以某公司大楼的目的选层电梯系统为背景。

目的选层电梯系统的运行流程：

  1. 进入电梯前，乘客输入目标楼层；
  2. 系统根据设定的策略指定最合适的电梯来服务，将结果告知乘客；
  3. 乘客进入指定电梯后，无需再次按楼层按钮，电梯会自动将其送至目标楼层。

在三次迭代作业中，你需要根据需求与正确性约束，设计多线程程序来模拟该电梯系统的运行。

系统从标准输入中读入乘客请求信息（起点层、终点楼层），请求调度器会根据此时电梯运行状态将乘客请求合理分配给某部电梯。被分配请求的电梯会经过上下行，开关门，乘客进入/离开电梯等动作将乘客从起点层运送到终点层。

请求的输入通过课程组提供的输入接口来定时投放请求，你需要调用课程组提供的接口进行输入输出。接口的具体细节见第四部分。

单部电梯的运行策略与电梯系统的调度策略均可任意指定。也就是说，在任意时刻，电梯的上下行动、开关门操作都可以自定义，只要保证在电梯系统运行时间不超过题目要求时间上限的前提下将所有的乘客送至目的地即可。电梯运行的具体介绍见第三部分。

第三部分：电梯运行（改动）

功能概述

对于每个乘客请求，需要调度电梯将其完成，并将必要的运行信息通过输出接口进行输出。具体而言，你需要控制电梯上下行，开关门的动作以及控制乘客进出电梯来将乘客从起点层运送到终点层。

在控制电梯运行的过程中需要满足所有系统运行约束，如运行速度、电梯载重等，运行约束的具体细节见本部分的系统运行约束一节。

在本次作业中:

  • 不指定电梯：仍然不为每个乘客请求指定电梯。你需要先通过输出 RECEIVE 信息表示乘客被分派给哪个电梯，然后调度指定的电梯来接送对应乘客。
  • 临时检修请求：与上一次作业要求一致。电梯会在正常运行时接收到临时检修请求，你需要合理安排当前未完成的乘客请求，并在限定时间内按规定流程调度电梯完成检修，在完成检修后电梯恢复正常运行。
  • （新增）双轿厢改造、双轿厢回收请求：本次作业中，每个电梯井道中初始时刻与前两次作业相同，只有一个主轿厢。此外每个井道有一个备用轿厢存放于F1层的暗室中。
    • 改造：电梯在正常运行时会接到双轿厢改造请求，你需要合理安排当前未完成的乘客请求，并在限定时间内按规定完成双轿厢改造，将位于暗室中的备用轿厢移动到电梯井中，完成改造后进入双轿厢模式运行。
    • 回收：电梯在双轿厢模式运行时会接到双轿厢回收请求，你需要合理安排当前未完成的乘客请求，并在限定时间内按规定完成双轿厢回收，将位于电梯井中的备用轿厢回收到暗室中，完成回收后恢复正常运行。
    • 电梯在双轿厢模式下运行时要满足相应规定。

电梯系统默认参数（改动）

电梯系统基础参数

  • 主轿厢可到达楼层：地下 B4-B1 层，地上 F1-F7 层，共 11 层
  • 主轿厢初始位置：F1 层（电梯井中）
  • 主轿厢数量：6 部
  • 编号：6 部主轿厢的 id 依次为 1-6
  • 移动速度：0.4s/层
  • 额定载重量：400kg
  • 开关门：开门到关门之间的间隔不小于 0.4s

双轿厢相关参数（新增）

  • 备用轿厢数量：6 部
  • 备用轿厢编号：6 部备用轿厢的 id 依次为 7-12，主轿厢 i 对应的备用轿厢编号为 i+6
  • 备用轿厢初始位置：F1 层（暗室中）
  • 换乘层：F2 层
  • 双轿厢模式下可到达楼层：
    • 主轿厢：F2-F7 层
    • 备用轿厢：B4-F2 层
  • 移动速度、额定载重量、开关门时间均与单轿厢模式一致

电梯载重说明

允许开门时暂时超过载重限制，只需保证关门时轿厢符合载重限制即可。电梯系统的全部电梯（包括 6 部主轿厢和 6 部备用轿厢）的载重均为 400kg。

RECEIVE约束说明

必须保证电梯在做出任何动作时都必须有未完成的 RECEIVE。需要在 RECEIVE 信息中输出每个乘客被分配给了哪个电梯运送。

临时检修

检修流程（微小改动）

本次作业中 NORMAL 状态不仅可能由上一次检修结束转移而来，还可能由上一次双轿厢回收结束转移而来。每次临时检修时，电梯状态都须按下表顺序进行转移 (NORMAL -> ACCEPT -> REPAIR -> TEST -> NORMAL)。

电梯状态 转移到此状态的标志 电梯行为流程 电梯行为额外约束
NORMAL
(默认状态)		 1. 初始状态
2. 输出 MAINT-END (上一次检修结束)
3. 输出 RECYCLE-END (上一次回收结束)		 1. 正常运行，直到收到指令
*2. 收到指令，自动输出 MAINT-ACCEPT		 无
REP_ACCEPT
(收到检修指令)		 程序输出 MAINT-ACCEPT 1. 运行到 F1 层，准备检修
2. 轿厢内普通乘客下电梯并关门
*3. 检修工进入，关门，输出 MAINT1-BEGIN		 1. 检修工进入时无普通乘客
2. 期间只允许关门一次，禁移/禁开
3. 必须处于关门状态
4. 见下方特殊约束
REPAIR
(部件检修状态)		 程序输出 MAINT1-BEGIN 1. 检修等待至少 T_stop = 1s
*2. 输出 MAINT2-BEGIN		 1. 此前所有 RECEIVE 结束
2. 静默状态 (禁移/禁开关门/禁出入)
3. 禁止 RECEIVE 新乘客
4. 须处于 REPAIR 状态
TEST
(验收测试状态)		 程序输出 MAINT2-BEGIN 1. 速度调为 0.2s/层
2. F1 -> 目标楼层 -> F1
3. 速度恢复 0.4s/层
4. F1层开门让工人下，关门
*5. 输出 MAINT-END		 1. 速度 0.2s/层
2. 须按规定路线移动
3. 仅在 F1 开门一次让工人下
4. 禁进新乘客，禁 RECEIVE
5. 须处于 TEST 状态

(注：带 `` 的步骤表示状态转移的标志)*

额外说明

  • REP_ACCEPT 状态特殊约束：进入该状态后电梯仍可自由运行，但检修响应时间 T_complete（输出 ACCEPT 到 END 之间的时间）不得超过 7s。
  • 检修工人：遵循普通乘客规则（开门上下等），固定起点/终点为 F1，无需输出 RECEIVE，不计入性能分。
  • RECEIVE 说明：MAINT1-BEGIN 强制结束当前所有 RECEIVE，程序需将未完成请求重新进行 RECEIVE 分配。

双轿厢改造与回收（新增）

同一电梯井的主轿厢和备用轿厢总是同时处于相同状态。

双轿厢改造流程 (NORMAL -> UP_ACCEPT -> UPDATE -> DOUBLE)

电梯状态 转移到此状态的标志 电梯行为流程 电梯行为额外约束
NORMAL 1. 初始状态
2. MAINT-END
3. RECYCLE-END		 1. 正常运行
*2. 收到改造指令，输出 UPDATE-ACCEPT		 无
UP_ACCEPT 程序输出 UPDATE-ACCEPT 1. 主轿厢到 F3 层，准备改造
2. 乘客下电梯，关门
*3. 输出 UPDATE-BEGIN		 1. 须处于关门状态
2. 见特殊约束
UPDATE 程序输出 UPDATE-BEGIN 1. 等待至少 T_stop = 1s
*2. 输出 UPDATE-END		 1. 此前 RECEIVE 结束
2. 禁新请求，处于静默
3. 须处于 UPDATE 状态
DOUBLE 程序输出 UPDATE-END 1. 备用轿厢移动到井中 F1 层
2. 双轿厢模式正常运行		 1. 主备均可 RECEIVE
2. 遵循双轿厢规则

双轿厢回收流程 (DOUBLE -> REC_ACCEPT -> RECYCLE -> NORMAL)

电梯状态 转移到此状态的标志 电梯行为流程 电梯行为额外约束
DOUBLE 程序输出 UPDATE-END 1. 正常运行
*2. 收到回收指令，输出 RECYCLE-ACCEPT		 无
REC_ACCEPT 程序输出 RECYCLE-ACCEPT 1. 备用轿厢驶向 F1 层
2. 乘客下电梯，关门
3. 主轿厢正常运行
*4. 输出 RECYCLE-BEGIN		 1. 见特殊约束
RECYCLE 程序输出 RECYCLE-BEGIN 1. 备用轿厢 RECEIVE 结束
2. 等待至少 T_stop = 1s
*3. 输出 RECYCLE-END		 1. 备用轿厢禁新请求并静默
2. 主轿厢在 F2 以上正常运行
NORMAL 程序输出 RECYCLE-END 1. 备用轿厢回暗室
2. 主轿厢恢复单厢模式		 1. 主轿厢恢复 B4-F7 运行
2. 备用轿厢静默停用

双轿厢运行规则

  • 主轿厢范围 F2 ~ F7，备用轿厢范围 B4 ~ F2。
  • 两轿厢不能同时处于同一楼层，严禁相撞。
  • 电梯每次输出 ARRIVE 时刻所处楼层即变为新楼层（不考虑运动中间态）。
  • （新增）双轿厢电梯可不受 RECEIVE 约束从换乘层 F2 移动一层以避让。

时间约束

  • 改造时间约束：UPDATE-ACCEPT 到 UPDATE-END 之间的时间不得超过 6s。
  • 回收时间约束：RECYCLE-ACCEPT 到 RECYCLE-END 之间的时间不得超过 6s。

系统运行约束（改动汇总）

  1. 基本约束：关门状态下移动；开门到关门间隔不小于 0.4s；运行结束时必须全部关门并送达所有乘客。
  2. RECEIVE 约束：
    • 乘客进电梯必须在输出该乘客的 RECEIVE 之后（中途下车需重新 RECEIVE）。
    • 电梯移动必须有未结束的 RECEIVE（避让 F2 层除外）。
    • MAINT1-BEGIN、UPDATE-BEGIN、RECYCLE-BEGIN 会强制结束对应轿厢的所有 RECEIVE。
    • 暗室中的备用轿厢禁止 RECEIVE。
  3. 载重：关门时必须 <= 400kg。

第四部分：输入/输出说明（改动）

请使用课程组提供的输入输出接口，严禁伪造时间戳。

输入格式

  • 乘客请求：[时间戳]乘客ID-WEI-体重-FROM-起点层-TO-终点层
  • 检修指令：[时间戳]MAINT-电梯ID-检修工人ID-验收目标楼层
  • 改造指令：[时间戳]UPDATE-主轿厢电梯ID
  • 回收指令：[时间戳]RECYCLE-备用轿厢电梯ID

输出格式

  • 接收分配：[时间戳]RECEIVE-乘客ID-电梯ID
  • 到达/开门/关门：[时间戳]ARRIVE/OPEN/CLOSE-所在层-电梯ID
  • 进/出电梯：
    • 进：[时间戳]IN-乘客ID-所在层-电梯ID
    • 出(到达目标)：[时间戳]OUT-S-乘客ID-所在层-电梯ID
    • 出(未达目标)：[时间戳]OUT-F-乘客ID-所在层-电梯ID
  • 检修信息：MAINT1-BEGIN / MAINT2-BEGIN / MAINT-END (带电梯ID)
  • 改造信息：UPDATE-BEGIN / UPDATE-END (带主轿厢电梯ID)
  • 回收信息：RECYCLE-BEGIN / RECYCLE-END (带备用轿厢电梯ID)

数据限制（改动）

  • T_max 计算：T_max = max(T_std + 30, 1.3 * T_std) (中测和强测严格限制该时间)
  • 互测数据：T_max 统一为 120s。
  • 保证系统结束时启用的备用轿厢都会被回收。
  • 保证在合法状态下才会收到检修/改造/回收指令。

第六部分：性能分

总分 100 = 85 (功能) + 15 (性能)。
性能分由系统运行时间 (T_run)、平均完成时间 (T_avg) 和系统耗电量 (W) 组成。

  • T_run = max(T_real, T_final)
  • T_avg = Σ(T_i - T_i0) / N
  • W = W_OPEN * N_OPEN + W_CLOSE * N_CLOSE + W_ARRIVE * N_ARRIVE
    (其中 W_OPEN = W_CLOSE = 0.1, W_ARRIVE = 0.4)

最终性能分公式为：
s = 15 * (0.3 * r(T_run) + 0.3 * r(T_avg) + 0.4 * r(W))
(其中 r(x) 为折算函数)

第七部分：实现提示

  1. 策略与执行解耦：建议编写一个等候乘客表类和一个电梯策略类（分析并返回目标动作）。
  2. 状态模式建模：到达 -> 开门 -> 进出 -> 关门 -> 寻层 -> 移动。
  3. 分配器：建议编写分派器线程作为后续作业的基础。
  4. 性能基准 (ALS 策略)：
    • 主请求：若空车，选最高优先级（最早到达）；若有人，选车内最高优先级。
    • 被捎带请求：投喂时间 <= 关门截止时间，且方向一致。
